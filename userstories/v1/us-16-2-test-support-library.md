# US-16.2: `UrkelTestSupport` — Test Helpers Library

## 1. Objective

Create a dedicated **`UrkelTestSupport`** Swift library that provides ergonomic test helpers for machines generated by Urkel — including mock client builders, transition recorders, path assertion helpers, and Swift Testing macros — making it trivially easy to write thorough, readable tests for any Urkel-generated state machine.

## 2. Context

The generated test stubs (US-13.4) give developers a scaffold, but they don't provide any infrastructure to make those tests easy to write. Developers currently have to:

- Manually construct `Client.testValue` with individual closure overrides for each transition.
- Manually track which transitions fired and in what order.
- Write bespoke assertion helpers for every machine.
- Use `withDependencies` from `swift-dependencies` for every test, which requires knowing the dependency key.

`UrkelTestSupport` eliminates this boilerplate with a library of composable test utilities that work with **any** Urkel-generated machine, regardless of its specific states or transitions. It integrates tightly with Swift Testing (`@Test`, `#expect`, `#require`) and `swift-dependencies`.

This library is imported only in test targets (`testTarget(.name("MyTests"), dependencies: ["UrkelTestSupport"])`), never in production code.

## 3. Acceptance Criteria

* **Given** `UrkelTestSupport` is imported in a test target, **when** a developer writes a test for a generated machine, **then** they can construct a fully stubbed `Client` with a single `MockClient(machine:)` builder that pre-fills all closures with sensible no-op or failable stubs.

* **Given** a `TransitionRecorder` attached to a `Client`, **when** transitions are fired during a test, **then** the recorder captures each `(from: String, event: String, to: String)` triple in order, and `#expect(recorder.transitions == [...])` reads like plain English.

* **Given** `#expectPath(machine:fires:endsIn:)` macro or function, **when** called with a sequence of events and an expected final state, **then** it drives the machine through the events and asserts the final state matches — in a single line.

* **Given** a machine with guards (US-12.1), **when** using `MockClient`, **then** guards are individually overridable via `MockClient { $0.isValidURL = true }` syntax, mirroring `withDependencies { $0.client = ... }`.

* **Given** a machine with `@invoke` (US-14.1), **when** using `MockClient`, **then** the invoked service closures are pre-stubbed to return a default value (or throw a canned error), and can be overridden per test without boilerplate.

* **Given** a machine with delayed transitions (US-14.2), **when** using `MockClient`, **then** the `clock` dependency is automatically set to `ImmediateClock` so timers fire synchronously in tests.

* **Given** a test that exercises only a subset of transitions, **when** `MockClient` is used in strict mode, **then** any closure that was never called causes the test to fail with a "unexpected call to [closureName]" message — catching over-broad implementations.

* **Given** `UrkelTestSupport` is used alongside the generated test stubs (US-13.4), **when** a developer fills in a stub, **then** they can call `withUrkelDependencies(client: MockClient(...)) { ... }` as a one-liner to scope the client to the test body.

## 4. Implementation Details

* **New SPM target** in `Package.swift`:
  ```swift
  .target(
      name: "UrkelTestSupport",
      dependencies: [
          .product(name: "Dependencies", package: "swift-dependencies"),
          .product(name: "Testing", package: "swift-testing"),  // if available
      ]
  )
  ```

* **`TransitionRecorder`**:
  ```swift
  public final class TransitionRecorder: Sendable {
      public struct Entry: Equatable, Sendable {
          public let from: String
          public let event: String
          public let to: String
      }
      public private(set) var transitions: [Entry] = []
      public func record(from: String, event: String, to: String) { ... }
      public func reset() { ... }
  }
  ```
  Integrated with the runtime trace protocol (US-13.5): attach a recorder to the `events: AsyncStream<UrkelEvent>` from the client.

* **`MockClientBuilder<Client>`** — a generic builder that:
  1. Starts with `Client.testValue` as the base.
  2. Provides a chainable API for overriding individual closures.
  3. Attaches a `TransitionRecorder` automatically if tracing is enabled.
  4. Injects `ImmediateClock` for any client that has a `clock` property.

* **`#expectPath` helper function** (not a macro initially, can be upgraded):
  ```swift
  public func expectPath<InitialState, FinalState>(
      _ machine: consuming Machine<InitialState>,
      fires events: [(Machine<InitialState>) async throws -> Any],
      endsIn finalStateType: FinalState.Type,
      sourceLocation: SourceLocation = #_sourceLocation
  ) async throws
  ```

* **`withUrkelDependencies`** — thin wrapper around `withDependencies` that accepts a pre-built `MockClient`:
  ```swift
  public func withUrkelDependencies<Client: DependencyKey, R>(
      _ client: Client.Value,
      operation: () async throws -> R
  ) async rethrows -> R
  ```

* **Strict mode** — `MockClient(strict: true)` wraps each unoverridden closure in an `XCTFail` / `Issue.record` call so unexpected calls surface immediately.

* **Documentation** — each public API gets a DocC article with a before/after example showing the reduction in test boilerplate.

## 5. Testing Strategy

* Unit-test `TransitionRecorder`: records entries in order; reset clears entries; concurrent appends are safe (`actor` or `Mutex`).
* Unit-test `MockClientBuilder`: default build produces `testValue`; overrides apply correctly; strict mode fails on uncalled closure.
* `#expectPath`: correct path passes; wrong final state fails with clear message; intermediate error reported at correct step.
* `ImmediateClock` injection: machine with `after(30s)` settles synchronously in test.
* Integration: write tests for `BluetoothScale` using `UrkelTestSupport`; compare line count before/after.
* DocC: all public symbols have documentation; at least one tutorial article in the DocC catalog.
