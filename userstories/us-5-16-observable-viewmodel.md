# US-5.16: Observable ViewModel Generation

## Objective

Optionally generate an `@Observable` view-model class that bridges a
machine's `AsyncStream` output events to `@Observable` properties, giving
SwiftUI callers a zero-boilerplate reactive layer without embedding
`@Observable` into the generated machine itself.

## Background

US-5.15 generates `AsyncStream` for output events — the correct
abstraction for the state-machine layer. But SwiftUI developers typically
want `@Observable` properties that update the view automatically, not
`for await` task loops they write by hand.

The right architecture is:

```
CounterMachine  ──AsyncStream──►  CounterViewModel  ──@Observable──►  SwiftUI
  (generated,                       (generated or                      body
   ~Copyable,                        user-authored)
   Sendable)
```

The view-model sits between the machine and the UI. It:
1. Holds the `CounterClient` dependency.
2. Drives the machine through its lifecycle.
3. Listens to `outputs` and maps events to `@Observable` properties.
4. Exposes transition trigger methods (`func start()`, `func finish()`, …).

Generating this class is **optional** (opt-in via `urkel-config.json`
`"generateViewModel": true`) because:
- It requires macOS 14 / iOS 17 (Observation framework).
- Teams may prefer their own architecture (TCA, MVI, Redux, …).
- The machine alone is sufficient for non-UI use cases.

## Input DSL (same counter from US-5.15)

```
machine Counter

@states
  init Idle
  state Active
  final Done

@entry Active / resetCount

@transitions
  Idle   -> start  -> Active
  Active -*> increment       / incrementCount
  Active -*> decrement       / decrementCount
  Active -*> value(count: Int)                # output event → @Observable
  Active -> reset  -> Active
  Active -> finish -> Done
```

## Generated Output (fourth file: `CounterViewModel.swift`)

```swift
// Generated by Urkel — do not edit.
// Source: counter.urkel
// Requires: macOS 14+ / iOS 17+  (Observation framework)

import Dependencies
import Foundation
import Observation

// MARK: - Counter ViewModel

/// @Observable bridge between CounterMachine and SwiftUI.
///
/// Drives the state machine lifecycle and maps output events to
/// observable properties. Inject `CounterClient` via `@Dependency`.
///
/// ## Usage
/// ```swift
/// @State private var vm = CounterViewModel()
///
/// var body: some View {
///     Text("\(vm.count)")
///     Button("Start") { Task { await vm.start() } }
///     Button("Increment") { Task { await vm.increment() } }
///     Button("Finish") { Task { await vm.finish() } }
/// }
/// ```
@Observable
@MainActor
public final class CounterViewModel {

    // MARK: - Observable state

    /// Latest count emitted by the machine. `nil` before first emission.
    public private(set) var count: Int? = nil

    /// Current lifecycle phase (mirrors the machine's phase at runtime).
    public private(set) var phase: CounterViewPhase = .idle

    // MARK: - Dependency

    @ObservationIgnored
    @Dependency(\.counter) private var client

    // MARK: - Private

    @ObservationIgnored
    private var _state: CounterState?

    // MARK: - Lifecycle

    public init() {}

    // MARK: - Trigger methods (one per consuming transition)

    /// Transition from `Idle` → `Active`.
    public func start() async {
        guard case .idle = phase, let machine = client.makeObserver() as? CounterMachine<CounterPhase.Idle>? else { return }
        let active = await machine.start()
        phase = .active
        _startObserving(active)
    }

    /// Transition from `Active` → `Active` (re-enters, fires @entry).
    public func reset() async {
        // handled internally via state forwarding
    }

    /// Transition from `Active` → `Done`.
    public func finish() async {
        guard case .active = phase else { return }
        phase = .done
    }

    // MARK: - In-place triggers

    /// Sends `increment` to the machine (no-op outside `Active`).
    public func increment() async { }

    /// Sends `decrement` to the machine (no-op outside `Active`).
    public func decrement() async { }

    // MARK: - Output stream observation

    private func _startObserving(_ machine: consuming CounterMachine<CounterPhase.Active>) {
        let stream = machine.outputs
        Task { [weak self] in
            for await event in stream {
                guard let self else { return }
                switch event {
                case .value(let count):
                    self.count = count
                }
            }
        }
    }
}

// MARK: - CounterViewPhase

/// Observable mirror of the machine's phase, safe to use in SwiftUI.
public enum CounterViewPhase: Sendable {
    case idle
    case active
    case done
}
```

> **Note**: The view model's trigger methods and in-place bridges are
> generated as stubs — the emitter fills in the machine wiring at code-
> generation time based on the `.urkel` source. The stub above is for
> illustration; the final generated form is complete.

## Configuration

`urkel-config.json`:

```json
{
  "generateViewModel": true,
  "viewModelSuffix": "ViewModel"   // default; produces CounterViewModel
}
```

The `generateViewModel` flag defaults to `false`. When `true`, Urkel emits
a fourth file `XxxViewModel.swift` alongside the three standard files.

The `UrkelPlugin` propagates this config to the CLI via
`--generate-view-model`, and the `UrkelGenerate` command plugin does the
same.

## Why a separate file (not inline in the machine)?

- Keeps the machine target free of `Observation` framework dependency
  (older platform support, server-side Swift).
- The view-model can be regenerated independently without touching the
  machine — cleaner diffs.
- Teams that don't want the view model simply set `"generateViewModel": false`.

## Acceptance Criteria

* **Given** `"generateViewModel": true` in config, **when** `urkel generate`
  is run, **then** a `CounterViewModel.swift` file is produced alongside the
  three standard files.

* **Given** the view model file, **when** compiled on macOS 14+, **then** no
  compiler errors.

* **Given** `vm.start()` is called, **when** the live code calls
  `emitValue(42)`, **then** `vm.count` becomes `42` on the `@MainActor`.

* **Given** `vm.finish()` is called, **when** polled after the call,
  **then** `vm.phase == .done` and the output stream observation task has
  ended.

* **Given** `"generateViewModel": false` (default), **when** `urkel generate`
  is run, **then** no `XxxViewModel.swift` is produced — other three files
  are unaffected.

* **Given** a machine with no output events, **when** view model is generated,
  **then** it compiles without the `_startObserving` and stream loop (they
  are omitted).

## Testing Strategy

* Config unit test: `generateViewModel: true` produces 4 URLs; `false`
  produces 3.
* Snapshot test: `CounterViewModel.swift` for the counter fixture.
* Compile test: view model file compiles cleanly (part of
  `generatedSwiftCompiles` test suite).
* `@MainActor` annotation is present on the class — verified by string
  inspection in snapshot.
